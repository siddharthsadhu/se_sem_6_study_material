Introduction to Software Engineering

Process Framework · Software Myths · Process Models · Agile Development · Team Structure

What is Software Engineering? CORE

Software Engineering is the application of a systematic, disciplined, quantifiable approach to the development, operation, and maintenance of software. (IEEE Definition)

Simple Definition (Pressman) Software engineering is the systematic design and development of software products AND the management of the software process.

What is Software?

Software is not just programs — it consists of three components:

💻

Instructions

Computer programs that when executed provide desired features, functions, and performance.

🗄️

Data Structures

Enable the programs to adequately manipulate information.

📄

Descriptive Information

Hard copy and virtual forms that describe the operation and use of the programs.

Science vs Engineering

Key Distinction Scientists "build in order to learn," while engineers "learn in order to build." SE is an engineering discipline focused on solving real-world problems.

Why Study Software Engineering?

  • Economies of ALL developed nations depend on software
  • Software costs often dominate computer system costs
  • Software costs more to maintain than to develop — maintenance can be several times the development cost
  • Concerned with cost-effective software development

Elements of Software Complexity (Brooks)

Famous Quote "The complexity of software is an essential property, not an accidental one." — Brooks
  • Complexity of the Problem Domain — Requirements are hard to pin down
  • Complexity of the Development Process — Managing teams, tools, change
  • Complexity in Software Itself — Internal logic and interactions
  • Behavior of Discrete Systems — Non-continuous, hard to test exhaustively
The task of a software development team is to engineer the illusion of simplicity.

Process Framework & Activities

What is a Software Process?

A process defines Who is doing What, When and How to reach a certain goal.

Process = Collection of activities, actions, and tasks performed when a work product is created.

Generic Process Framework Activities (CPDCDE)

Communication

Communicate with stakeholders to understand objectives and gather requirements.

Planning

Define tasks, resources, timelines, risks, and scheduling.

Modelling

Create models of requirements and design to better understand the software.

Construction

Code generation and testing of the software.

Deployment

Deliver software to customers; gather feedback.

Process Hierarchy

Activity → Action → Task An Activity strives to achieve a broad objective. An Action produces a major work product. A Task focuses on a small, well-defined objective with a tangible outcome.

Generic Process Models (SDLC)

1. Waterfall (Plan-Driven / Cascading) Model

Waterfall Model — Cascading Phases
✅ Advantages
  • Organized approach; robust phase separation
  • Reflects common engineering practice
  • Easy to understand and use
❌ Disadvantages
  • Doesn't cope well with changing requirements
  • Working version available only late
  • Development teams may wait for each other
Best Used When Requirements are well known and few changes are likely.

2. Incremental Model

Incremental Model
✅ Advantages
  • Early delivery of parts of the system
  • Lower risk of project failure
  • Reduces rework in construction
❌ Disadvantages
  • Increments need to be relatively small
  • Mapping requirements to increments not easy

3. Evolutionary (Rapid Prototyping) Model

Involves building a prototype quickly, showing it to the client, refining based on feedback, and repeating until the final product is produced.

✅ Advantages
  • Deals constantly with changes
  • Provides quick initial version
  • Good for unclear requirements
❌ Disadvantages
  • Quick fixes may be involved
  • "Invisible" process — not well-documented
  • Client may think first version is close to final

4. Spiral Model (Meta-Model) — by Boehm

EXAM IMPORTANT The Spiral Model is called a "Meta-Model" because it encompasses all other process models. It was proposed by Barry Boehm. The primary determinant in each spiral is Risk.
Spiral Model — 4 Quadrants

Q1 — Objectives

Functionality, Performance, HW/SW Interfaces, Critical Success Factors, Constraints (Cost, Schedule)

Q2 — Risk Analysis

Identify & resolve risks; build prototype; evaluate alternatives

Q3 — Development

Develop and verify next-level product; includes design, coding, testing

Q4 — Planning

Plan next phases; review results; commit to next iteration

Spiral is called Meta-Model because it can incorporate all other models (waterfall, prototyping, incremental) within its iterations.

5. Rational Unified Process (RUP)

RUP is an iterative and incremental software development framework. It describes how to effectively implement 6 best practices:

  • Develop Iteratively
  • Manage Requirements
  • Use Component Architectures
  • Model Visually
  • Verify Quality
  • Control Changes
RUP Phases & Disciplines

6. Component-Based Software Engineering (CBSE)

Builds systems by integrating pre-built, reusable software components.

✅ Advantages
  • Reduces software to be developed in-house
  • Allows faster delivery
  • More reliable (pre-tested components)
❌ Disadvantages
  • Compromises in requirements needed
  • Less control over system's evolution

Software Myths
The VASA Ship Story — A Viking ship that sank after less than 1 nautical mile due to: faulty specifications, ignored QA, requirement creep, excessive schedule pressure, changing needs, and lack of technical specs. Analogy for bad SE practices.

Lessons from VASA

  • Accountability: Who is responsible when a product is flawed at launch?
  • Stick to plans: Maintain discipline once a plan is set
  • Independent reviews: Incorporate independent reviewers to avoid bias
  • Quality monitoring: Identify problems early through continuous monitoring
  • Test early and often
Agile Development MANIFESTO

Agile Manifesto was created in 2001 by the Agile Alliance. It emerged from methods like XP, SCRUM, DSDM, Adaptive Software Development, Crystal, and Feature-Driven Development.

Agile Manifesto — 4 Core Values

Individuals & Interactions

Over processes and tools — Research shows 50× increase in performance through good communication

Working Software

Over comprehensive documentation — Post deployment defects cost 100× more

Customer Collaboration

Over contract negotiation — Dedicated product owner → 4× productivity increase

Responding to Change

Over following a plan — 64% of waterfall features are rarely or never used

Agile Development is based on: Iterative and Incremental development

Benefits of Individuals & Interactions

  • Communication is clear, effective, and quick
  • Teamwork becomes strong as people work together
  • Development teams can self-organize
  • More chances to innovate and customize processes
  • Deeper job satisfaction for team members

Software Team Structure

Functional Organization

Dedicated departments (HR, IT, Sales). Functional Manager controls team. Maximum flexibility in use of staff.

Project-ized Organization

Everyone works on a project and reports to Project Manager. PM has direct control. Strong identity and commitment.

Matrix Organization

Team member reports to BOTH Functional Manager and Project Manager. Balanced/Strong/Weak variants.

Matrix Types Balanced Matrix: Equal authority. Strong Matrix: PM has authority over FM. Weak Matrix: FM has authority over PM.

Understanding Requirements CORE
Why Requirements Matter (Brooks) "The hardest single part of building a software system is deciding precisely what to build."

Definition of Requirement (IEEE)

  • A condition or capability needed by a user to solve a problem or achieve an objective
  • A condition that must be met by a system or component to satisfy a contract, standard, or specification
  • A documented representation of the above

Stakeholders & Traceability

Stakeholders: Anyone with interest in the system — clients, end users, developers, managers, domain experts.

Good vs Bad Requirements

✅ Good Requirements
  • Unambiguous — only one interpretation
  • Verifiable — can be tested
  • Consistent — no contradictions
  • Complete — no missing information
  • Traceable — can be linked to design
❌ Bad Requirements
  • Ambiguous — multiple interpretations
  • Non-verifiable — cannot be tested
  • Contradictory — inconsistent
  • Incomplete — missing information

Software Design

Design Concepts · Architecture · Modularity · Patterns · UI Design · SDS

Design Fundamentals
What is Software Design? Deriving a solution that satisfies software requirements — including Problem Understanding, Identifying Solutions, Describing Solution Abstractions, and Repeating until design is expressed in primitive terms.

Why Design?

  • Large systems decomposed into sub-systems providing related services
  • Architectural design establishes the framework for sub-system control and communication
  • Output is a description of software architecture

Advantages of Design

System Analysis

Making architecture explicit early to meet critical requirements (performance, reliability, maintainability)

Stakeholder Communication

High-level presentation used as focus for discussion by different stakeholders

Large-Scale Reuse

Systems with similar requirements developed using large-scale software reuse

Design Phases

1. Architectural Design

Identify sub-systems and their relationships

2. Abstract Specification

Specify each sub-system's services and constraints

3. Interface Design

Describe how each sub-system interfaces with others

4. Component Design

Decompose sub-systems into components

5. Data Structure Design

Design structures to hold problem data

6. Algorithm Design

Design algorithms for problem functions

Key Design Concepts

Modularity

Divide system into discrete modules. Easier to understand, test, and maintain.

Abstraction

Focus on essential characteristics, hide details. Different levels of abstraction.

Separation of Concerns

Each module addresses a specific concern. Reduces complexity.

Information Hiding

Module's internal implementation hidden from others. Only interfaces exposed.

Cohesion

How closely related the operations in a module are. HIGH cohesion is desirable.

Coupling

Degree of interdependence between modules. LOW coupling is desirable.

Architectural Patterns & Styles

Choosing Architecture Depends On:

  • Performance: Localize critical operations; minimize communications
  • Security: Use layered architecture with critical assets in inner layers
  • Safety: Locate safety-related features in few sub-systems
  • Availability: Include redundant components; design for fault tolerance
  • Maintainability: Use fine-grain, replaceable components

System Organization Styles

1. Repository Model

Repository (Blackboard) Architecture
  • All shared data held in a central database
  • Efficient for sharing large amounts of data
  • Sub-systems that produce data need not know how data is used

2. Client-Server Model

  • Servers offer services; Clients call on these services
  • A network allows clients to access services
  • Easy to add new servers; upgrade servers transparently
  • Effective use of networked/distributed systems

3. Layered Model (Abstract Machine Model)

Layered Architecture
  • Supports incremental development
  • Changeable and portable architecture
  • Each layer provides services to the layer above

UI Design — Golden Rules EXAM IMPORTANT
Interface Design Fact System users often judge a system by its interface rather than its functionality. A poorly designed interface can cause catastrophic errors.

The Three Golden Rules (Schneiderman)

👤

1. Place the User in Control

  • Define interaction modes that don't force undesired actions
  • Provide flexible interaction
  • Allow interaction to be interruptible and undoable
  • Streamline interaction as skill levels advance
  • Hide technical internals from casual users
  • Design for direct interaction with objects on screen
🧠

2. Reduce User's Memory Load

  • Reduce demand on short-term memory (≤7 items)
  • Establish meaningful defaults
  • Define intuitive shortcuts
  • Visual layout based on real-world metaphors
  • Disclose information in a progressive fashion
🔄

3. Make the Interface Consistent

  • Allow user to put current task in meaningful context
  • Maintain consistency across family of applications
  • Don't make changes without compelling reason if users have expectations

Design Principles (Sommerville)

Human Factors in Interface Design

  • Limited short-term memory: People can instantaneously remember about 7 items
  • People make mistakes: Inappropriate alarms increase stress and likelihood of more mistakes
  • People are different: Wide range of physical capabilities; different interaction preferences

UI Design Process

UI design is an iterative process involving close liaisons between users and designers. The 3 core activities:

  • User Analysis: Understand what users will do with the system
  • System Prototyping: Develop a series of prototypes for experiment
  • Interface Evaluation: Experiment with prototypes with users

Web App Design — Content Design

  • Aesthetic Design: Visual appeal, color harmony, typography
  • Content Design: Quality, clarity, and relevance of content
  • Architecture Design: Overall structure and navigation
  • Navigation Design: How users move through the system
Software Design Specification (SDS)

SDS documents the complete software design solution. It includes:

  • Architectural design description
  • Data design (data structures, database schema)
  • Interface design (UI wireframes, API specs)
  • Component/procedural design (algorithms, flow)

Software Quality Assurance (SQA)

SQA is a planned and systematic pattern of all actions necessary to provide adequate confidence that a software product conforms to established technical requirements.

SQA Tasks

  • Apply technical methods and tools
  • Conduct formal technical reviews
  • Enforce standards and procedures
  • Control changes
  • Measure and report

Cost of Quality

Prevention Costs

Spent BEFORE building: establishing methods/procedures, training, acquiring tools, quality planning.

Appraisal Costs

Spent AFTER building but BEFORE shipping: inspection, testing, reviews.

Internal Failure Costs

Defects found BEFORE delivery: rework, retesting, redesign.

External Failure Costs

Defects found AFTER delivery: complaints, repairs, returns, warranty costs.

Six Sigma for SE

Six Sigma is a quality management methodology aiming for 3.4 defects per million opportunities. Key process: DMAIC (Define, Measure, Analyze, Improve, Control).

Software Testing

Objectives of Testing

  • A successful test uncovers a yet undiscovered error
  • A good test is not redundant; is "best of breed"; neither too simple nor too complex

Testing Levels

V-Model — Verification & Validation Levels

Testing Methodologies

White-Box Testing (Structural Testing)

Also known as: Clear box testing, Glass box testing, Transparent box testing, Structural testing. Requires knowledge of internal program design and code.

Tests based on coverage of code statements, branches, paths, conditions.

  • Function Coverage: Has each function been called?
  • Statement Coverage: Has each statement been executed?
  • Branch Coverage: Has each branch of control structures been executed?
  • Condition Coverage: Has each boolean sub-expression evaluated both true and false?

Techniques: API testing, Code coverage, Fault injection, Mutation testing, Static testing

Black-Box Testing (Functional Testing)

No knowledge of internal design required. Tests based on requirements and functionality. Only aware of what software should do, not how.

Techniques:

  • Equivalence Partitioning: Divide input into classes that should behave the same
  • Boundary Value Analysis: Test at boundaries of input partitions
  • State Transition Tables: Test system behavior across state changes
  • Decision Table Testing: Test combinations of conditions
  • All-pairs Testing: Test all possible discrete combinations

Errors found: Incorrect/missing functions, Interface errors, Data structure errors, Performance errors, Initialization/termination errors

Verification & Validation (V&V)

Verification

"Are we building the product RIGHT?"

Evaluating software to determine whether products of a given phase satisfy conditions imposed at the start of that phase. Checks conformance to specification.

Types: Dynamic (testing during execution) and Static (code review, walkthroughs)

Validation

"Are we building the RIGHT product?"

Checks that the product design satisfies the intended usage. Ensures the product meets actual user needs and specifications were correct in the first place.

Types: Data validity, Logical validity

V&V Process: Verification and Validation continue in a cycle until the software becomes defects-free.

Software Testing Lifecycle (STLC) Phases

1. Requirements Study

Study client requirements; define test objectives and coverage; plan schedule, standards, resources

2. Test Case Design & Development

Component identification; test specification design; test specification review

3. Test Execution

Code review; test execution and evaluation; performance and simulation

4. Test Closure

Test summary report; project de-brief; project documentation

5. Test Process Analysis

Analyze what went well and what didn't; capture lessons learned

LOC & Function Point Metrics

Size-Oriented Metrics — LOC (Lines of Code)

Simplest and most widely used metric. Comments and blank lines should NOT be counted.

Productivity = Output / Effort
Errors per KLOC (Thousands of Lines of Code)
Defects per KLOC · $ per LOC
✅ Advantages
  • Simple to measure
  • Widely used and understood
  • Easy to automate counting
❌ Disadvantages
  • Varies with coding style
  • Focuses on coding only
  • Penalizes high-level languages and code reuse
  • Measures lexical complexity only
  • Correlates poorly with quality and efficiency

Effort & Productivity Calculations

Effort = Number of People × Time (in person-days/months)

Productivity = Software Size (LOC) / Effort

Cost = Effort × Cost per person-month
Example: 4 developers, 15 days → Effort = 4 × 15 = 60 person-days. If cost/day = ₹3000 → Total Cost = ₹1,80,000. If size = 24,000 LOC and effort = 12 person-months → Productivity = 2000 LOC/person-month.

Function Point Analysis (FP)

Function Points measure systems from a functional perspective, independent of technology/language/platform.

Components of Function Points

ComponentAbbreviationDescription
External InputsEIData crossing boundary from outside to inside
External OutputsEOData crossing boundary from inside to outside
External InquiriesEQInput + output components resulting in data retrieval
Internal Logical FilesILFUser-identifiable groups of logically related data maintained inside the system
External Interface FilesEIFUser-identifiable groups of logically related data maintained outside the system

FP Calculation Formula

FP = UFC × [0.65 + 0.01 × ΣFi]

Where:
UFC = Unadjusted Function Count (sum of all weighted components)
Fi = Value Adjustment Factors (14 factors, each rated 0-5)

Example: Fi = 46 → FP = 50 × [0.65 + 0.01 × 46] = 50 × 1.11 = 56

14 Value Adjustment Factors (VAF)

Data Communication · Distributed Data Processing · Performance Objectives · Transaction Rate · On-line Data Entry · End-User Efficiency · Heavily Used Configuration

On-line Update · Complex Processing · Reusability · Operational Ease · Multiple-Site Use · Facilitate Change · Conversion & Install Ease

Applications of FP

  • Estimate project cost, duration, and staffing size
  • Predict number of errors during testing
  • Forecast number of source lines in the implemented system
  • Determine productivity benefits of new tools

Magic Formulas (FP to Schedule)

Project Duration = 2.5 × ∛(Work Months)
Optimum Staffing Size = √(Work Months)
Minimum Project Duration = 0.75 × ∛(Work Months)

Example: 27 work months → Duration = 2.5 × 3 = 7.5 months · Staff = √27 ≈ 5.2

Cyclomatic Complexity (McCabe)

Used to evaluate complexity of an algorithm. Count of test cases needed to test a method comprehensively.

V(G) = E – N + 2
Where E = number of edges, N = number of nodes

Higher V(G) = More complex = More test cases needed
Example: A sorting algorithm flow graph with E=10, N=8 → V(G) = 10 - 8 + 2 = 4 (4 independent test paths needed)

Risk Management EXAM IMPORTANT
What is Risk? "A risk is a potential future harm that may arise from some present action." Risk = expectation of loss. Caused by lack of information, control, or time.

Risk Categorization

Project Risks

Threaten the project PLAN. If real: schedule slips, costs increase.

Technical Risks

Threaten QUALITY and timeliness. If real: implementation becomes difficult or impossible.

Business Risks

Threaten VIABILITY of software. Sub-categories: Market, Strategic, Sales, Management, Budget risks.

Risk Types by Knowledge

  • Known Risks: Uncovered after careful evaluation of project plan (e.g., unrealistic delivery date)
  • Predictable Risks: Extrapolated from past project experience (e.g., past turnover)
  • Unpredictable Risks: Can occur but extremely difficult to identify in advance

Reactive vs Proactive Risk Strategies

⚠️ Reactive (Fire Fighting)
  • "Don't worry, I'll think of something"
  • Nothing done until something goes wrong
  • Team flies into action to correct rapidly
  • Crisis management = management approach
✅ Proactive (Risk Management)
  • Risk management steps are followed
  • Primary objective: avoid risk
  • Have contingency plan for unavoidable risks
  • Controlled and effective manner

Seven Principles of Risk Management

  1. Maintain a global perspective (view risk in context of system and business problem)
  2. Take a forward-looking view (think about future risks; establish contingency plans)
  3. Encourage open communication (all stakeholders can point out risks at any time)
  4. Integrate risk management (into the software process)
  5. Emphasize a continuous process of risk management
  6. Develop a shared product vision (facilitates better risk identification)
  7. Encourage teamwork when managing risk

Part II — 3-Mark Answers

Q: Software doesn't wear out. Then what leads to software failures?

Answer: Unlike hardware, software does not physically wear out — it doesn't degrade due to environmental factors or mechanical wear. However, software does deteriorate in a logical sense:
  • Change-induced failures: Every time software is modified to meet new requirements or fix bugs, new defects can be introduced (the "change curve")
  • Increasing complexity: Each patch adds complexity that was not in the original design
  • Requirement changes: Software designed for one environment may fail in a changed environment
  • Design flaws: Errors in initial design that surface later during use
  • Undiscovered bugs: Defects that exist from the beginning but are not triggered until specific conditions occur
  • Technology obsolescence: As hardware and OS evolve, old software may behave unexpectedly
This is why software requires preventive maintenance (predictive maintenance) — to restructure and improve before failures occur.

Q: What are Functional and Non-Functional Requirements?

Functional Requirements define WHAT the system should do — specific behaviors, functions, data processing, and outputs. They describe system features and services (e.g., "The system shall allow users to log in with email and password").

Non-Functional Requirements define HOW WELL the system performs — quality attributes and constraints. They measure performance, reliability, security, usability, maintainability, and portability (e.g., "The system shall load pages within 2 seconds under 1000 concurrent users").

Key difference: FR = "what to do"; NFR = "how well to do it." NFRs are often harder to verify and tend to apply to the system as a whole rather than individual features.

Part II — 6-Mark Answers

Q: Explain Spiral Model in detail. Why is it called a Meta-Model?

Spiral Model (Boehm, 1986)

The Spiral Model is an iterative software development process model that combines elements of both design and prototyping, incorporating risk analysis in each iteration. It was proposed by Barry Boehm.

Structure — 4 Quadrants per Spiral Iteration:
  1. Q1 — Determine Objectives, Alternatives, Constraints: Establish objectives (functionality, performance), identify alternatives (build, reuse, sub-contract), identify constraints (cost, schedule)
  2. Q2 — Evaluate Alternatives, Identify & Resolve Risks: Analyze each alternative against constraints, identify and assess risks, build prototypes to resolve risks
  3. Q3 — Develop and Verify: Design, code, test, and verify the next-level product based on resolved risks
  4. Q4 — Plan Next Iteration: Review results with stakeholders, commit resources for next spiral iteration

Why is it called a Meta-Model? The Spiral Model is called a Meta-Model because it can encompass all other process models within its framework. When the risk analysis in Q2 identifies that waterfall is best for the current iteration, waterfall is used. If prototyping is better, a prototype is built. If the incremental approach suits, that's applied. It is a model OF models — each loop of the spiral can use a different SDLC approach based on risk assessment. This generality and flexibility make it a "meta" (higher-level) model.

Advantages: Best for large, risk-heavy projects · Handles changing requirements · Risk-driven approach · Combines best features of other models
Disadvantages: Complex to manage · Requires risk expertise · Can be expensive · Not suitable for small projects

Q: Draw the stages of RUP Model.

Rational Unified Process (RUP)

RUP has 4 phases and multiple disciplines. Disciplines have varying intensity across phases:

Phase 1 — Inception: Define scope; establish business case; identify risks; establish feasibility. Milestone: Life Cycle Objectives (LCO).

Phase 2 — Elaboration: Analyze problem domain; establish architectural foundation; eliminate highest-risk elements. Milestone: Life Cycle Architecture (LCA).

Phase 3 — Construction: Develop all components and features; integrate into product; test all functions. Milestone: Initial Operational Capability (IOC).

Phase 4 — Transition: Transfer product to users; beta testing; performance tuning; user training. Milestone: Product Release (PR).

Disciplines (run across all phases): Requirements · Analysis & Design · Implementation · Test · Deployment · Configuration & Change Management · Project Management · Environment

RUP best practices: Develop Iteratively · Manage Requirements · Use Component Architectures · Model Visually · Verify Quality · Control Changes
